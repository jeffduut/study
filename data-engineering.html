Lesson 1

The Fundamental Why

Core Idea: Data engineering exists because raw data, in its natural state, is useless.

Every company, every system, every human activity generates data. A click on a website. A sensor reading from a factory floor. A transaction at a checkout. A GPS ping from a delivery truck. In its raw form, this data is scattered, inconsistent, messy, and often meaningless.

The goal of data engineering is to turn raw data into trusted, accessible, usable data — data that people and systems can reliably act on.

The Three Ultimate Goals

Everything a data engineer does serves one or more of these three goals:

  1. Availability — Make data exist where it needs to be, when it needs to be there. If data isn't present and ready, nothing else matters.
  2. Reliability — Make data trustworthy. People must believe the data is correct, complete, and consistent. If they don't trust it, they won't use it — or worse, they'll use it and make bad decisions.
  3. Accessibility — Make data understandable and usable by the people and systems that need it. Data that exists and is correct but is impossible to query or interpret is still useless.

The Data Engineering Venn Diagram

Think of data engineering as the intersection of three domains:

The Three Lenses ┌─────────────────────┐ │ SOFTWARE ENG │ │ (code, systems, │ │ automation) │ └──────────┬──────────┘ │ ┌──────────┴──────────┐ │ DATA ENGINEERING │ │ │ ┌────┴─────┐ ┌────────┴────┐ │ DOMAIN │ │ DATA & │ │ KNOWLEDGE│ │ STATISTICS │ │ (business│ │ (quality, │ │ context)│ │ pipelines) │ └──────────┘ └─────────────┘
  • Software engineering — building automated, reliable, maintainable systems.
  • Data & statistics — understanding data shapes, distributions, quality metrics, and transformation logic.
  • Domain knowledge — understanding what the data means in the real world and what questions people will ask of it.

The Fundamental Constraint

Data engineering operates under a constant tension between volume, velocity, variety, and veracity (often called the "4 V's"). Every decision a data engineer makes is a trade-off among these forces:

DimensionWhat it meansThe challenge
VolumeHow much data there isStoring, moving, and processing large quantities efficiently
VelocityHow fast data arrivesKeeping up with the speed of production
VarietyHow many different forms data takesHandling diverse formats, schemas, and sources uniformly
VeracityHow trustworthy the data isDetecting and correcting errors, inconsistencies, and gaps
Mental Model Summary: Data engineering = making data available, reliable, and accessible in the face of difficult volume, velocity, variety, and veracity.

Case Study 1 — Spotify's Move to Reliability

Case Study: Spotify (2015–2017)
The Situation: Spotify's early data infrastructure was a collection of scripts run ad-hoc by individual engineers. Data analysts didn't trust the numbers because different scripts produced different results for the same question. A weekly "number of active users" report could vary by 5–15% depending on which script was run.

The Problem: All three goals were failing. Data was available (it existed somewhere) but not reliable (inconsistent definitions, no guarantees) and not accessible (only the engineer who wrote a script could interpret its output).

The Response: Spotify invested in a proper data platform team, defined company-wide metric definitions, built automated testing for data pipelines, and created a unified system for transforming raw event data into trusted tables. They shifted from "move data somehow" to "deliver trustworthy data products."

Takeaway: Reliability is not a luxury. When data can't be trusted, no one uses it, and the entire investment in collecting data is wasted.

The Core Mental Model: The Pipeline

Core Idea: Every data engineering system can be understood as a pipeline: Source → Ingestion → Storage → Transformation → Serving. This is the single most important mental model in the discipline.

No matter how complex a data system appears, it reduces to five sequential stages. Every tool, every architecture, every decision fits into one of these boxes.

The Universal Data Pipeline ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ │ SOURCE │───→│INGESTION│───→│ STORAGE│───→│TRANSFORM│───→│ SERVING│ │ │ │ │ │ │ │ │ │ │ │ Where │ │ How we │ │ Where │ │ How we │ │ How it │ │ raw │ │ bring │ │ we keep│ │ clean, │ │ gets │ │ data │ │ it in │ │ it │ │ join, │ │ consumed│ │ lives │ │ │ │ │ │ aggr. │ │ │ └────────┘ └────────┘ └────────┘ └────────┘ └────────┘

The Five Stages Explained

  1. Source — Where the data originates. A database, an API, a file upload, a sensor, a user action. The data engineer typically has no control over the source; it is whatever the business produces.
  2. Ingestion — The process of moving data from the source into the engineering system. This is where questions of frequency (real-time vs. batch), volume, and reliability first appear.
  3. Storage — Where data is kept. The engineer chooses the storage paradigm (row-oriented, column-oriented, document, blob) that matches how the data will be used.
  4. Transformation — The work of turning raw stored data into clean, structured, meaningful data. This is where most of the engineering effort lives. Cleaning, validating, joining, aggregating, computing.
  5. Serving — The final stage where data is delivered to its consumers: dashboards, reports, machine learning models, APIs, or other applications.

The Fundamental Insight

Every stage feeds the next. If a stage breaks, everything downstream breaks. The strength of a data pipeline is the strength of its weakest stage. A brilliantly engineered transformation layer is useless if ingestion silently drops 10% of the data.

Hand Exercise 2.1

✏️ EXERCISE — Map a Familiar System

Example answer:

The Pipeline Mindset

Once you see every data problem as a pipeline problem, you gain a superpower: you can diagnose any issue by asking "which stage is failing?"

  • Data is missing? → Ingestion or source problem.
  • Data is slow? → Storage or transformation problem.
  • Data is wrong? → Transformation or source problem.
  • Data is correct but no one can get to it? → Serving problem.
Mental Model Summary: The pipeline is the universal pattern. Any data system, no matter how complex, decomposes into Source → Ingestion → Storage → Transformation → Serving. Learn to see every problem through this lens.

Case Study 2 — LinkedIn's Data Pipeline Evolution

Case Study: LinkedIn (2010–2015)
The Situation: In LinkedIn's early days, data processing was done via a collection of ad-hoc scripts running directly against production databases. Each team pulled data the way they wanted. The production databases were frequently slowed to a crawl by these data queries.

The Problem: The pipeline stages were not separated. Source (production databases) and serving (analytics queries) were collapsed together. Queries from the serving stage competed with live user traffic at the source stage.

The Response: LinkedIn built a dedicated data pipeline. First, they decoupled ingestion (moving data out of production databases into a separate system). Then they built a transformation layer (cleaning, joining, aggregating the data into a data warehouse). Finally, they created a serving layer that analysts could query without touching production systems.

Takeaway: Physically and logically separating the five pipeline stages protects each stage from interfering with the others. This is not an academic exercise — LinkedIn's production databases were breaking under the load of unconstrained data access.

Lesson 3

Data Sources & The Raw Reality

Core Idea: Data sources are chaotic, ungoverned, and varied. The first act of data engineering is accepting that you cannot control the source — you can only adapt to it.

Data engineers do not create data (usually). They receive it. And what they receive is never clean, never consistent, and never documented as well as they would like.

The Three Categories of Sources

CategoryExamplesCharacteristic Challenge
Transactional / Operational User signups, orders, payments, inventory changes High volume of small, structured records. Schema changes over time. Must capture every change without loss.
Event / Behavioral Clickstream, app usage, sensor readings, server logs Massive volume. Semi-structured. Timestamp-critical. Often "fire and forget" — if you miss it, it's gone.
External / Third-Party Weather data, market data, partner feeds, social media APIs No control over format, frequency, or reliability. APIs change without notice. Data may have licensing constraints.

Source Realities You Must Accept

  1. Sources drift. The data format that arrives today may not be the format that arrives tomorrow. Fields get added, renamed, removed. Values that were once required become optional (or vice versa).
  2. Sources are unreliable. An API goes down. A database replica lags. A file transfer times out. The source has no "responsibility" to the data engineer — it has its own job to do.
  3. Sources have no guarantees. The source team did not design their system with your pipeline in mind. They will not notify you when they make changes. Your job is to build resilience against this.
  4. Sources produce duplicates. Many source systems (especially transactional databases with change logs like PostgreSQL WAL or MySQL binlog) provide exactly-once guarantees within their own transaction boundaries. But when your ingestion system reads from them, it may re-read the same transaction if it crashes mid-batch. At the point of ingestion, you must always assume duplicates are possible. Build your pipeline to handle this.

The First Engineering Decision: Push vs. Pull

Every source must be accessed in one of two ways:

  • Pull — Your system reaches out and fetches data from the source. You control when and how often. But you must know what to fetch and when new data exists.
  • Push — The source sends data to you. You don't control timing, but you don't need to. You must be ready to receive at any time.

Case Study 3 — Uber's Source Diversity Crisis

Case Study: Uber (2014–2016)
The Situation: Uber's rapid global expansion meant that every new city introduced new data sources. Different cities used different payment processors. Different regulatory environments required different data formats. The engineering team was drowning in source variety.

The Problem: Every new source required a custom ingestion script. There were hundreds of scripts, all maintained by different people. When a source changed (a payment processor updated their API), the corresponding script broke silently, and data was lost for days before anyone noticed.

The Response: Uber invested in a schema-on-read approach and a unified event specification (called "Uber's Event Taxonomy"). Instead of building custom ingestion for every source, they defined a standard schema for common event types (trip events, payment events, etc.) and required all sources to map their data to these schemas at the point of ingestion. They also built automated monitoring that detected schema drift and alerted engineers.

Takeaway: Variety, left unchecked, becomes an exponential maintenance burden. The solution is not to control sources but to build a consistent abstraction layer that sources adapt to at the point of entry.

Ingestion & The Arrival Problem

Core Idea: Ingestion is the most fragile stage. It is the first point of failure, and failures here are often silent. The goal of ingestion is not just to move data, but to move data reliably and verifiably.

Ingestion is the bridge between the messy, uncontrolled world of sources and the controlled world of your pipeline. The core tension in ingestion is between speed and completeness.

The Two Great Modes

Batch IngestionStreaming / Real-time Ingestion
WhenAt scheduled intervals (every hour, every night)Continuously, as events happen
Volume per unitLarge, predictable chunksSmall, unpredictable trickle or surge
Error handlingRetry the whole batchMust handle each record individually
ComplexityLower (simpler to build and reason about)Higher (ordering, exactly-once, state management)
Best forHistorical reports, large bulk loads, non-time-critical dataDashboards, alerts, real-time features, time-sensitive decisions

The Ingestion Guarantees

Every ingestion system makes a promise (implicitly or explicitly) about how it handles data delivery. There are exactly three levels:

  1. At-most-once: Each record is delivered zero or one times. If something fails, the record is lost. Fast, but data may disappear.
  2. At-least-once: Each record is delivered one or more times. No data loss, but duplicates may appear downstream.
  3. Exactly-once: Each record is delivered exactly one time. The ideal, but the hardest to achieve. Requires coordination between the source and the ingestion system.

In practice, achieving true end-to-end exactly-once is difficult. Some modern systems (e.g., those using transaction logs, checkpointed state, and idempotent sinks) can deliver genuine exactly-once. However, many practical systems operate at at-least-once and use downstream deduplication with careful state management to approximate exactly-once semantics. Downstream deduplication is not a silver bullet — it requires fault-tolerant state storage, careful handling of window boundaries, and awareness that deduplication state can itself be lost on crash. The key trade-off: exactly-once has higher complexity and latency; at-least-once is simpler but imposes work on downstream stages.

The Silent Failure Problem

The most dangerous failure in ingestion is silent failure — when data is lost but the system reports success. This can happen when:

  • A record is malformed and quietly skipped
  • A network timeout drops a batch but the acknowledgment is sent anyway
  • A schema mismatch causes records to be written to a dead-letter queue that nobody monitors
  • A source changes its output format and your parser silently returns nulls

Late-Arriving Data

One of the most painful real-world problems is late-arriving data — records that arrive hours, days, or even weeks after the event they describe actually occurred. A mobile app user's session data may be buffered on their device and only uploaded when they reconnect to WiFi. A store's batch upload may be delayed by a snowstorm.

Late-arriving data creates a fundamental tension between completeness and timeliness. If you wait for all possible late data, your reports are never current. If you publish reports on a fixed schedule, you will miss late records. Common strategies include:

  • Watermarking: Declare a cutoff time after which you stop accepting data for a given window. Data that arrives after the watermark is assigned to the next window. This is common in streaming systems.
  • Reconciliation: Publish preliminary results on schedule, then revise them when late data arrives. The dashboard shows "preliminary" until the reconciliation window closes.
  • Idempotent upserts: Design your storage so that late-arriving records can update previously published aggregates without double-counting. This requires partition-level idempotency and careful merge logic.

The choice depends on your consumers: a fraud detection system cannot wait for late data (it needs decisions now), but a quarterly financial report can wait for reconciliation.

Case Study 4 — Airbnb's Ingestion Pipeline Failures

Case Study: Airbnb (2015–2017)
The Situation: Airbnb's data pipeline ingested events from millions of user interactions every day. They relied on a batch ingestion system that pulled logs from servers every hour.

The Problem: The ingestion system had no end-to-end validation. It tracked whether files were transferred, but it did not track whether the records within those files were valid. Over time, several silent failures accumulated: server logs were being truncated during rotation (missing the last 5 minutes of events), some events were being written in a new format that the parser silently skipped, and network congestion was causing some files to arrive empty.

The Impact: Data science teams were building models on incomplete data. A search ranking model was trained on data that was missing 12% of booking events, causing it to underweight certain user behaviors. The gap was undetected for six months.

The Response: Airbnb built what they called "data-integrity monitoring" — a system that tracked expected record counts per source per time window. Any deviation from the expected range triggered an alert. They also implemented a "data contract" system where each source had to declare its schema, and any deviation from the declared schema caused the batch to be quarantined rather than silently consumed.

Takeaway: Ingestion is not done when the file arrives. Ingestion is done when the data is verified. Always build a "did we get what we expected?" check. A silent ingestion failure is worse than no data at all — at least with no data, you know you have a problem.

Storage & The Shape of Data

Core Idea: Storage is not neutral. How you store data determines what you can do with it. The shape you choose is a bet on how the data will be queried in the future.

Every storage system makes a trade-off between how data is written (ingested) and how it is read (queried). In general, optimizing for one comes at the expense of the other, though some advanced systems (LSM-tree-based designs) achieve strong performance on both axes through techniques like compaction and bloom filters.

The Two Fundamental Shapes

Row-Oriented vs. Column-Oriented Storage ROW-ORIENTED (transactional, many columns at once): ┌──────────────────────────────────────────────┐ │ Row 1: user_1 | bob@... | signup | 2024-01-01│ │ Row 2: user_2 | alice@..| login | 2024-01-01│ │ Row 3: user_3 | carol@..| logout | 2024-01-01│ └──────────────────────────────────────────────┘ Best when: you need all columns for one record at a time. COLUMN-ORIENTED (analytics, few columns at once): ┌──────────┬───────────────┬─────────┬────────────┐ │ user_1 │ user_2 │ user_3 │ ... │ │ bob@... │ alice@... │ carol@..│ ... │ │ signup │ login │ logout │ ... │ │ 2024-.. │ 2024-.. │ 2024-.. │ ... │ └──────────┴───────────────┴─────────┴────────────┘ Best when: you aggregate values across millions of rows (e.g., "average signups per day").

The Five Storage Questions

When evaluating a storage choice, ask these five questions in order:

  1. Is it durable? — If the power goes out, is the data gone? (The most important question. If data isn't durable, nothing else matters.)
  2. Is it queryable? — Can I ask questions of this data without moving it elsewhere?
  3. Is it scalable? — Will it still work when data volume grows 10x? 100x?
  4. Is it cost-effective? — Storage is cheap, but querying is expensive. The total cost includes how much you pay to access the data, not just to store it.
  5. Is it compatible? — Can my ingestion and transformation systems talk to it?

The Data Lake vs. Data Warehouse Distinction

In modern data engineering, you will hear two terms constantly. Here is the pure distinction stripped of technology marketing:

Data LakeData Warehouse
PurposeStore everything in its original formStore clean, structured data for analysis
SchemaSchema-on-read (interpret at query time)Schema-on-write (validate before storing)
Data qualityRaw — may be messy, duplicated, incompleteCurated — clean, consistent, documented
UsersData engineers, data scientists (who do their own exploration)Analysts, business users, dashboards
Best analogyA shipping container port — everything arrives, we sort it out laterA library — everything is catalogued and organized before it arrives

Most mature data organizations have both: a data lake for raw, exploratory data, and a data warehouse (populated from the lake via transformation) for trusted, high-performance analytics.

The Lakehouse: A Unified Approach

In recent years, the strict binary between data lake and data warehouse has softened with the emergence of the lakehouse architecture. A lakehouse adds warehouse-like features (ACID transactions, schema enforcement, concurrent read/write support) directly on top of the data lake using open table formats. This eliminates the need to maintain two separate systems while giving engineers the best of both worlds: the flexibility of a lake and the reliability of a warehouse.

Storage Formats Matter

The way data is physically encoded on disk has a massive impact on performance and cost. There are three dominant formats in data engineering:

  • Row-oriented formats (Avro, JSON, CSV): Data is stored row by row. Good for write-heavy workloads, streaming ingestion, and cases where you need all columns for each record. Compression ratios are generally lower because different data types are mixed within each row.
  • Columnar formats (Parquet, ORC): Data is stored column by column. Excellent for analytics queries that read only a subset of columns across many rows. Columnar storage achieves much higher compression (similar data types compress better) and enables predicate pushdown (skipping entire row groups that don't match filters). Best for read-heavy analytical workloads.
  • Hybrid / semi-structured (Avro with schema evolution): Avro is particularly good for scenarios where schema evolution is frequent, as the schema is stored alongside the data.

The choice of format directly affects query performance, storage cost, and schema evolution capabilities. A well-designed pipeline may use multiple formats: Avro for streaming ingestion (fast writes), Parquet for analytical storage (fast reads).

Partitioning: Organizing Data for Query Performance

Partitioning is the practice of physically dividing data into separate storage locations based on a key (usually a date, but could be a region, a customer segment, etc.). When a query filters on the partition key, the storage system can skip entire partitions — this is called partition pruning and can reduce query costs by orders of magnitude.

  • Date partitioning is the most common scheme: data is stored in folders like year=2024/month=01/day=15/. A query for a single day reads only one partition instead of scanning all data.
  • Multi-level partitioning uses multiple keys: e.g., region=US/year=2024/month=01/. Each additional level further prunes the search space, but too many levels creates too many small files (the "small file problem").
  • The cardinality trade-off: Partitions should have enough data to make scanning efficient, but not so much that pruning doesn't help. A rule of thumb: aim for partitions of 100 MB to 1 GB each.

Case Study 5 — Netflix's Storage Architecture Evolution

Case Study: Netflix (2012–2018)
The Situation: Netflix's data engineering team managed all their analytics data in a traditional row-oriented data warehouse. As Netflix grew globally, data volume exploded — billions of streaming events, ratings, search queries, and A/B test results per day.

The Problem: Row-oriented storage became prohibitively slow and expensive for analytics queries. A report on "percentage of users who complete a series within 7 days" required scanning terabytes of data to read just two columns (user_id, watch_time). The warehouse was spending 95% of its I/O reading and discarding columns it didn't need.

The Response: Netflix migrated their analytics workloads to a column-oriented storage layer. They adopted a "big data lake" architecture where raw data was stored in its original form (allowing data scientists to explore freely), and a curated columnar layer served the high-performance analytics queries they ran daily. The separation reduced query times from hours to minutes for the same reports.

Takeaway: Choosing the wrong storage shape for your workload is not a minor inefficiency — it can make entire classes of queries infeasible. Storage decisions are structural and hard to undo. Always model your query patterns before choosing your storage shape.

Transformation & Making Data Useful

Core Idea: Transformation is where raw data becomes useful data. It is the highest-effort, highest-value stage of the pipeline. Most of a data engineer's time is spent here.

Data in its raw form is rarely usable. It needs to be cleaned, validated, combined, restructured, and enriched. This is the transformation stage. It is also where the engineer's understanding of the domain matters most.

The Three Types of Transformation

  1. Structural transformations: Changing the shape or format of data without changing its meaning. Examples: converting JSON to rows, pivoting a table, splitting a column, filtering out irrelevant fields.
  2. Cleaning transformations: Fixing or removing data that is wrong. Examples: removing duplicate records, imputing missing values, correcting malformed dates, standardizing units (e.g., lbs to kg).
  3. Business logic transformations: Computing new information from raw data. Examples: calculating lifetime value from transactions, assigning a customer to a tier based on their spending, determining the "first session" for a new user.

The Fundamental Transformation Pattern: ETL/ELT

There are two ways to order the transformation process relative to storage. This is one of the most important architectural decisions in data engineering.

The Transformation Golden Rule: Idempotency

A transformation must be idempotent. Running the same transformation on the same input data must always produce the same output. This is non-negotiable. If a transformation is not idempotent, you cannot recover from failures, reprocess historical data, or audit your results.

What Makes a Transformation Idempotent?

Idempotency is achieved through the output strategy you choose:

  • Overwrite (full refresh): The transformation writes to a known location and replaces whatever was there before. This is trivially idempotent — running it twice produces the same final state. Best for small datasets or dimensions.
  • Partition-level replacement: The transformation writes to a specific partition (e.g., date=2024-01-15). If it runs twice, it simply overwrites the same partition. This is the most common pattern for large-scale pipelines because it allows selective reprocessing of time ranges.
  • Upsert / Merge: The transformation inserts new rows and updates existing ones based on a key. This is idempotent only if the source data for a given key is deterministic — if the same key can have different values on different runs, upsert will produce different results.
  • Append with deduplication: The transformation appends new rows but adds a unique identifier. Downstream stages deduplicate by this identifier. This is idempotent at the append step but requires downstream deduplication logic to be idempotent as well.

Non-Idempotent Patterns to Avoid

  • Unconditional append without dedup: Every run adds new rows with no way to identify or remove duplicates.
  • Non-deterministic functions: Using functions like RANDOM(), NOW(), or CURRENT_TIMESTAMP in transformation outputs — running twice produces different values.
  • Stateful operations without checkpointing: Running totals or cumulative counts that depend on the previous run's output, not on the raw source data.
  • Read-your-writes dependencies: A transformation that reads its own previous output as input — re-running changes the input, so the output changes non-deterministically.

Structuring Transformations for Safe Re-Execution

The safest pattern is: always partition by date, always overwrite partitions, never read your own output. This means each transformation reads only from upstream raw or cleaned layers, and writes to a time-partitioned location. If a bug is found, you fix the transformation and re-run it on the affected date range. Downstream tables are rebuilt from scratch (or from the corrected partitions). This is the pattern used at every organization that does data well.

Schema Evolution in Transformations

Sources change their schemas over time. Fields are added, renamed, removed, or re-typed. A resilient transformation pipeline must handle this:

  • Backward-compatible changes: Adding a new column with a default value should not break existing transformations. Build your system to ignore unknown fields at the source and propagate them when possible.
  • Breaking changes: Renaming or removing a column requires a coordinated update between the source, the ingestion layer, and all downstream transformations. This is where a schema registry helps — it tracks schema versions and allows transformations to declare which version(s) they support.
  • Schema-on-read vs. schema-on-write: Schema-on-read (applying schema at query time) is more tolerant of schema drift. Schema-on-write (validating schema at write time) catches errors earlier but breaks more frequently. Most mature systems use a hybrid: schema-on-write at the ingestion boundary (catch problems early) and schema-on-read in the exploration/analytics layer (flexibility for ad-hoc queries).

Slowly Changing Dimensions (SCD)

A common transformation challenge is managing dimension tables that change over time — e.g., a customer changes their address or a product changes its category. There are three standard approaches:

  • SCD Type 1 (Overwrite): The old value is simply replaced with the new value. Simple, but you lose history. Best for attributes where history doesn't matter (e.g., customer's current phone number).
  • SCD Type 2 (Add row with date range): Each change creates a new row with a valid_from and valid_to date. You keep full history. A query at a specific point in time uses the row that was active at that moment. Best for audit-critical attributes.
  • SCD Type 3 (Add column): Track the current and previous value in separate columns. A middle ground — you can see the previous value, but no further back. Best for attributes where you only need "before and after."

Choosing the right SCD strategy is a domain decision, not a technical one. It depends on whether the business needs to query historical states.

Case Study 6 — Stripe's Data Transformation Journey

Case Study: Stripe (2014–2019)
The Situation: Stripe processed payments for thousands of businesses. Each business had its own data format, its own reporting needs, and its own definition of key metrics (e.g., "What counts as a successful payment?").

Each layer only depends on the layer below it. This made transformations testable, auditable, and maintainable.

Takeaway: Transformations must be organized in layers, with clear dependencies and clear ownership. A flat, monolithic transformation pipeline becomes a nightmare as complexity grows. Every transformation should have exactly one job and one output.

Lesson 7

Data Quality & Trust

Core Idea: Data quality is not a one-time check. It is a property of the entire system, and it must be continuously measured and maintained. If consumers don't trust the data, your pipeline doesn't exist.

Data quality is the most under-invested area in data engineering. Teams spend months building pipelines and days thinking about quality. The result is that data consumers (analysts, product managers, ML engineers) spend weeks manually verifying numbers before they use them.

The Five Dimensions of Data Quality

DimensionQuestionHow to Measure
Completeness Is all the data here? Count of expected records vs. actual records. Percentage of missing values in critical fields.
Uniqueness Are there duplicates? Count of duplicate keys. Ratio of unique IDs to total records.
Timeliness Is the data current enough? Age of the most recent record. Time between event occurrence and availability.
Validity Does the data conform to its schema and rules? Percentage of records passing format checks. Enum values within expected set.
Consistency Does the data agree with itself? Cross-table comparisons. Running totals vs. aggregated totals.

The Data Contract

The most powerful tool for maintaining data quality is the data contract — an explicit, documented agreement between the data producer (or the pipeline stage) and the data consumer about:

  • Schema: What fields exist, their types, and which are required vs. optional.
  • Semantics: What each field means. (Is "revenue" gross or net? Does "active user" mean logged in today or this week?)
  • Service Level Objectives (SLOs): How fresh the data will be, how complete, how available.

A data contract turns a vague expectation ("this should be right") into a measurable promise ("this table will be 99.9% complete within 30 minutes of the hour").

Failures Must Be Loud

A fundamental principle of data quality is that silent failures are unacceptable. A pipeline that produces no data is better than a pipeline that produces wrong data — because at least you know there's a problem. Every quality check should either pass or raise an alert that wakes someone up.

Automated Data Observability

Beyond explicit data contracts and manual checks, a mature data engineering system incorporates automated data observability — continuous statistical profiling and anomaly detection across all datasets. This includes:

  • Distribution monitoring: Track the statistical distribution of every column. If a column that normally has values 1–100 suddenly has values 1–10,000, something changed — either the source or the data quality.
  • Drift detection: Detect when a dataset's characteristics shift over time, which may indicate a silent schema change, a source behavior change, or a data quality degradation.
  • Freshness monitoring: Track the age of the most recent record in every table. Alert if data stops arriving.

Automated observability complements explicit data contracts: contracts define what should be true, and observability detects what is true. Both are necessary.

Data Lineage: Tracing Data from Source to Sink

An essential companion to data quality is data lineage — the ability to trace a piece of data from its source, through every transformation, to every downstream consumer. Lineage answers questions like: "Which source table produced this value?" "If I change this transformation, which dashboards will break?" "Which batch of data was affected by yesterday's ingestion bug?"

Lineage can be captured at different levels of granularity:

  • Table-level lineage: Table X is derived from Tables A, B, C via Transformation Y.
  • Column-level lineage: Column X.revenue is computed from A.price × B.quantity.
  • Row-level lineage: This specific output record was produced from these specific source records (useful for debugging but expensive to store).

Without lineage, every data quality investigation starts from zero. With lineage, you can instantly identify the blast radius of a failure and trace a suspicious value back to its origin. Build lineage capture into your transformation layer — it's a force multiplier for every quality check and debugging exercise.

Data Governance: Who Can Access What, and Why

Data governance is the system of policies and processes that control who can access what data, under what conditions, and for how long. It is the bridge between data engineering and legal/compliance requirements. Every data engineer must understand the basics:

  • Access control: Not everyone should see every dataset. Customer PII (personally identifiable information), financial data, and employee records require role-based access controls. The principle of least privilege applies: grant only the access needed for the job.
  • PII detection and masking: Sensitive fields (names, emails, SSNs, credit card numbers) must be identified and protected. Common techniques include masking (showing only the last 4 digits), hashing (for join keys), and excluding (removing altogether from derivative datasets).
  • Compliance drivers: Regulations like GDPR (right to deletion, data portability), CCPA (consumer privacy), and HIPAA (health data privacy) impose legal obligations on how data is stored, processed, and retained. A data engineer's pipeline must support these requirements — for example, being able to delete all data for a specific user on demand (GDPR right to erasure).
  • Data retention: Not all data needs to be kept forever. Define retention policies: raw logs may be kept for 30 days, aggregated reports for 7 years (for compliance), PII-bearing data for as long as the business purpose requires (and no longer). Automated lifecycle management deletes or archives data based on these policies.
  • Audit logging: Every access to sensitive data should be logged: who accessed it, when, what query they ran, what results they saw. This creates an accountability trail.

Case Study 7 — The Knight Capital Group: Where Software Engineering and Data Quality Intersect

Case Study: Knight Capital Group (2012)
The Situation: Knight Capital was one of the largest market-making firms on Wall Street, handling over 10% of all US stock trades. Their systems processed market data in real time to execute trades automatically.

The Problem: On August 1, 2012, a catastrophic chain reaction was triggered. A piece of legacy code that was supposed to have been removed during a software deployment was accidentally left running. This code had a flag variable that indicated whether certain data had been processed, but the flag was reused across different functions — a classic software engineering failure (improper code cleanup, insufficient code review) that cascaded into a data quality failure (the flag variable held wrong data, causing the system to misinterpret market data).

The Impact: In 45 minutes, Knight Capital lost $440 million — more than the company's entire annual revenue. The company was effectively destroyed; it had to be acquired in a fire sale days later. The root cause was not a complex algorithm failure — it was a deployment hygiene failure (legacy code not removed) that produced a data quality failure (the flag variable had the wrong value). This case lives at the intersection: poor software engineering practices created bad data, and the system had no data quality checks to catch it.

Takeaway: Data quality depends on the entire software lifecycle, not just pipeline checks. A code deployment mistake can corrupt data just as surely as a broken pipeline. This case illustrates that data engineering and software engineering are inseparable: you cannot have trustworthy data without trustworthy code deployment practices. Additionally, every system that drives automated decisions must have "kill switches" that stop processing when quality checks fail.

Lesson 8

Orchestration & The Living System

Core Idea: A data pipeline is not a static thing — it is a living system that must run, rerun, recover, and evolve over time. Orchestration is the art of managing this lifecycle.

Building each stage of the pipeline is only half the work. The other half is making the entire system run reliably over time — handling failures, managing dependencies, retrying when things go wrong, and ensuring that every morning, the reports are ready.

What Orchestration Does

An orchestration system performs four essential functions:

  1. Scheduling: Runs each pipeline stage at the right time. Source A must be ingested before it can be transformed. The daily report must run after all source ingestions are complete.
  2. Dependency management: Ensures that a stage runs only after all its upstream stages have completed successfully. If any upstream stage fails, downstream stages wait or are skipped.
  3. Failure handling: When a stage fails, the orchestration system decides what to do: retry (up to X times), alert, skip, or run a remedial action.
  4. Monitoring: Tracks the state of every run — success, failure, duration, data volume — and surfaces anomalies.

The Orchestration Mental Model: The Directed Acyclic Graph (DAG)

Every pipeline can be represented as a graph where nodes are stages and edges are dependencies. The graph must be acyclic — no stage can depend on itself, directly or indirectly. This is non-negotiable.

A Simple DAG for a Daily Report Pipeline ┌──────────┐ │ START │ └────┬─────┘ │ ┌────────────┼────────────┐ ▼ ▼ ▼ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Ingest │ │ Ingest │ │ Ingest │ │ Payments │ │ Users │ │ Inventory│ └────┬─────┘ └────┬─────┘ └────┬─────┘ │ │ │ └────────────┼────────────┘ ▼ ┌──────────────┐ │ Transform │ │ (Join & │ │ Aggregate) │ └──────┬───────┘ ▼ ┌──────────────┐ │ Serve to │ │ Dashboard │ └──────────────┘

The DAG gives you two superpowers:

  • Parallelism: Independent stages run simultaneously (all three ingest jobs in the example above).
  • Isolation: A failure in one branch does not affect independent branches. Payments ingestion can fail without affecting inventory ingestion.

The Three Hardest Problems in Orchestration

  1. Backfilling: When you change a transformation, you need to re-run it on historical data. Orchestration must support re-processing arbitrary time ranges without destroying the system. There are two approaches:
    • Full refresh: Discard all existing output and recompute it from scratch. Simple but expensive — it re-processes data that hasn't changed. Best for small datasets or when the transformation logic has fundamentally changed.
    • Incremental backfill: Re-process only the affected time range and replace just those partitions. Requires date-partitioned outputs and idempotent transformations. Best for large datasets — re-processing 2 days of data instead of 2 years.
    The key to safe backfilling is partition-level idempotency: each output partition is independently replaceable. Common pitfalls include: double-counting (because the backfill overlaps with already-processed data), missing data (because the backfill window is too narrow), and cascading downstream updates (because changing a source table requires rebuilding every derived table).
  2. Idempotency at scale: Every DAG node must produce the same output for the same input, regardless of how many times it runs. If node T1 runs successfully on a 5-hour input window, then the orchestration system crashes, then T1 runs again on the same 5-hour input, it must produce the exact same output (not, say, double-count the data).
  3. Observability: When something fails, you must know what failed, why it failed, and what data was affected. This is harder than it sounds because failures cascade and the root cause may be three stages upstream from the first visible symptom.

Case Study 8 — Google's Orchestration of Borg

Case Study: Google (2004–2015, the Borg era)
The Situation: Google runs one of the world's largest data processing systems. Every day, trillions of events are ingested, and thousands of data pipelines process them — web crawl updates, ad metrics, search quality evaluations, spam detection model training.

The Problem: Without orchestration, individual teams managed their own dependencies. One team's pipeline would start processing before another team's data was ready. Failed runs were not retried automatically. When a source system was delayed, downstream pipelines all broke in different ways, and it took hours to figure out which data was stale and why.

The Response: Google built (and later published insights about) their orchestration system, which managed the entire data processing lifecycle as a DAG. Every data source had a declared "ready time." Every pipeline stage declared its dependencies. The orchestrator would not start a stage until all its dependencies had reported success. If a stage failed, it was retried with exponential backoff. If the retries exhausted, the DAG stopped and an alert was sent with the exact lineage of the failure.

Takeaway: Without orchestration, a data pipeline is a fragile collection of scripts that breaks regularly and is hard to reason about. With orchestration, it becomes a managed system with clear states, clear dependencies, and clear failure modes. The DAG is not a documentation artifact — it is the execution blueprint.

The Complete Picture: How It All Fits Together

Core Idea: The five stages, quality, and orchestration are not separate concerns. They form a unified system. A data engineer's job is to design, build, and maintain this system so that it delivers trustworthy data reliably over time.

Let us now assemble everything into a single coherent model.

The Unified Mental Model

Stages can branch, merge, and have conditional paths. The five stages are logical categories — an actual pipeline may have many parallel paths through them.

Broader Architectures: Data Mesh and Beyond

The unified model presented here assumes a centralized pipeline. In practice, many large organizations adopt a data mesh architecture, where each business domain owns its data pipelines as a self-serve product, and a shared infrastructure layer provides cross-cutting tools (storage, orchestration, governance, catalog). The data mesh does not replace the five-stage mental model — it distributes it across domain teams while centralizing the infrastructure that every domain needs. Whether you build a single unified pipeline or a mesh of domain-owned pipelines, the same stages and concerns apply.

The Data Engineer's Decision Tree

When approaching any data engineering problem, ask these questions in order:

  1. What are the sources? — What type of data? Push or pull? How reliable is the source?
  2. What are the consumers' needs? — How fresh must the data be? How complete? What shape do they need it in?
  3. What is the ingestion strategy? — Batch or streaming? At-least-once or exactly-once? How do we verify success?
  4. What is the storage strategy? — Data lake, warehouse, or both? Row or column oriented? Schema-on-read or schema-on-write?
  5. What transformations are needed? — Structural, cleaning, business logic? ETL or ELT? Layered or flat?
  6. How is quality maintained? — What are the data contracts? What checks run at each stage? What SLOs are promised?
  7. How is the system orchestrated? — What is the DAG? How are failures handled? How does backfilling work?
  8. How are metadata and governance managed? — Where is lineage tracked? How do consumers discover datasets? How are schemas versioned? How is access controlled? What compliance requirements apply?

This decision tree can be applied to any scenario, from a single CSV file that needs to end up in a dashboard, to a multi-billion-event-per-day global pipeline.

Properties of a Well-Designed System

A mature data engineering system exhibits these characteristics:

  • Observable: Anyone can look at any stage of the pipeline and see its state, volume, and latency.
  • Recoverable: Any stage can be re-run on any historical window without side effects.
  • Evolvable: Adding a new source or changing a transformation does not require rewriting the entire pipeline.
  • Traceable: Every value can be traced back to its source through documented lineage. If a number on a dashboard looks wrong, you can find its origin in minutes, not days.
  • Governed: There is a clear owner for every dataset, access controls for sensitive data, documented data contracts between stages, and compliance with relevant regulations.
  • Resilient: Failures at one stage do not cascade indefinitely. Downstream stages wait or receive explicit missing-data signals.
  • Economical: The cost of storage and compute is proportional to the value the data provides. This means using the right storage tier for each dataset (hot data on fast storage, cold data on cheap storage), limiting expensive cross-stage data movement, and ensuring that cost per query is visible to consumers. data engineering is often 30–60% of cloud costs at large organizations — cost awareness is a core competency, not an afterthought.

The real-time alert system uses in-memory state for the vitals threshold checks (cannot depend on disk I/O latency).

Transformations: T1 — Standardize: unify the five EHR schemas into a common patient-encounter model. Map each vendor's field names and codes to a hospital-wide standard. T2 — Clean: remove duplicate patient records (patients seen at multiple hospitals), impute missing values (e.g., missing weight uses last known value), flag implausible vitals (heart rate of 0). T3 — Join: combine EHR data with billing, scheduling, and vitals into a consolidated encounter record. T4 — Compute metrics: wait time = check-in time minus appointment time. Bed occupancy = occupied beds / total beds per hospital per shift. T5 — De-identify: strip PHI (name, DOB, SSN), generalize dates (month/year only), assign research IDs for the research mart.

Serving: Four outputs. (1) Real-time alert: a streaming system that evaluates every vital event against thresholds and sends a page to the assigned nurse's station. (2) Daily dashboard: a pre-aggregated table refreshed every morning, served to a BI dashboard that each hospital's admin team accesses. (3) Research dataset: a weekly snapshot of the de-identified warehouse, delivered to researchers with a data use agreement on file. (4) Financial report: a monthly aggregation of billing data joined with cost data, served to the CFO's office.

Data Quality: Data contracts for every source: EHR must have encounter_id (unique, not null), patient_id, timestamp, and at least one diagnosis code. ICU monitor must have device_id, patient_id, timestamp, and vital_type. Completeness monitor: every 15-minute batch from each hospital's EHR must contain at least 80% of the expected encounters (based on historical averages). Freshness SLO: real-time alerts within 30 seconds, daily dashboard data no older than T+6 hours, research dataset produced by the 5th of each month. Every quality failure sends an alert to the on-call engineer. The paper-scan pipeline has a special quality step: a human must validate a random 5% sample of scanned records before they are committed.

Orchestration: Two DAGs. DAG #1 (continuous): ICU monitor stream flows through a real-time processing graph — validate → compute threshold → alert. No retries (vitals are ephemeral; if an event is lost, the next event arrives). DAG #2 (batch): runs daily — pull EHR/billing/scheduling → standardize → clean → join → aggregate → publish to dashboard. Each hospital's EHR pull is an independent node (parallel). If Hospital A's EHR ingestion fails, the DAG continues for Hospitals B–E, and A's data is picked up in the next run with a backfill marker. Backfill support: any transformation can be run on any historical date range, and all outputs are partitioned by date. This allows re-processing after schema changes or bug fixes.

The Master Case Study — The Large Hadron Collider at CERN

Master Case Study: CERN Large Hadron Collider (2008–present)

Context: The Large Hadron Collider (LHC) at CERN is arguably the most demanding data engineering challenge in human history. It is a real-world existence proof that the mental model in this tutorial scales to the extreme.

The Scale:

  • The LHC produces about 1 billion proton collisions per second.
  • Each collision produces about 1 megabyte of data from the detectors.
  • That is 1 petabyte per second of raw data.
  • Total data produced per year: approximately 50 petabytes.
  • Total data stored across all LHC experiments (as of 2023): over 700 petabytes.

The Source: Four main detectors (ATLAS, CMS, ALICE, LHCb), each a different type of source with different data characteristics. Detectors produce raw electrical signals from particle collisions. These signals must be digitized, calibrated, and interpreted. Each detector produces data in different formats, at different rates, with different noise characteristics.

Ingestion — The Trigger System: The LHC cannot possibly store all 1 billion collisions per second. The ingestion system (called the "trigger") is a multi-stage filter:

  • Level 1 (hardware): Custom electronics decide in microseconds whether a collision is "interesting" (e.g., had enough energy, produced certain particles). Reduces 1 billion events/sec → 100,000 events/sec.
  • Level 2 (software): Fast algorithms reconstruct basic event properties. Reduces 100,000/sec → 10,000/sec.
  • Level 3 (software): Full event reconstruction for the most promising events. Final write rate: about 1,000 events/sec — a reduction factor of 1,000,000:1.

This is the ultimate example of the fundamental ingestion tension between speed and completeness. The LHC cannot store everything. It must make real-time decisions about what to keep. The cost of discarding a potentially important event is infinite (it's gone forever).

Storage — The Tiered Model: CERN uses a global tiered storage architecture:

  • Tier 0 (CERN, Geneva): Raw data from detectors. Tape storage. Massive, slow, cheap.
  • Tier 1 (13 major computing centers worldwide): Copies of raw data + reconstructed data. High-performance disk + tape.
  • Tier 2 (~140 universities and institutes): Processed data for analysis. Disk storage. Accessible to thousands of researchers worldwide.
  • Tier 3 (individual researchers' computers): Analysis datasets. Local storage.

This is the data lake and warehouse concept taken to global scale. Raw data lives at Tier 0/1 (the data lake). Processed, analyzed data lives at Tier 2/3 (the data warehouse/mart). Data moves between tiers based on usage patterns — hot data moves to faster storage, cold data stays on tape.

Transformation — The Reconstruction Pipeline: The transformation pipeline at CERN is one of the most complex in existence:

  • Raw data → Digitized: Convert detector signals to digital readings (charge, time, position).
  • Digitized → Reconstructed: Use algorithms to turn detector readings into particle trajectories and energies. This step alone involves terabytes of calibration data and runs on thousands of CPUs.
  • Reconstructed → Analyzed: Apply physics models to identify particles (electrons, muons, Higgs bosons, etc.) and compute high-level quantities (invariant masses, decay angles, etc.).

The reconstruction pipeline is re-run every time calibration data improves. This is backfilling at the petabyte scale. A new calibration model might require re-processing a year's worth of data — 50 petabytes.

Data Quality: CERN has rigorous data quality monitoring at every stage:

  • Detector quality: Each detector subsystem monitors its own data quality in real time. If a sensor starts producing noise, it is flagged and its data is marked as potentially bad.
  • Reconstruction quality: Every reconstructed event has quality flags. Events that fail quality cuts are excluded from analysis datasets.
  • Validation: Before any data is released to the collaboration, it passes through a validation pipeline that checks data integrity, calibration correctness, and consistency with previous data-taking periods.

The Higgs boson discovery (announced July 4, 2012) required data quality validation across five years of data, two different detectors, and thousands of scientists. Any undiscovered data quality issue would have invalidated the result. None was found.

Orchestration — The Worldwide LHC Computing Grid: CERN runs the largest distributed computing grid in the world (the WLCG). It orchestrates:

  • 170+ computing centers across 42 countries
  • Over 1 million CPU cores
  • Exabytes of storage
  • Thousands of data processing workflows, each with complex dependencies

The orchestrator (currently based on a system called DIRAC for LHCb, PanDA for ATLAS, CMS Workflow for CMS) manages the DAG of data processing: data arrives at Tier 0 → reconstruction jobs run → output is distributed to Tier 1s → analysis jobs run at Tier 2s. If a job fails, it is retried at a different site. If a site goes offline, jobs are redirected. The entire system runs 24/7/365 with minimal human intervention.

Lessons for Every Data Engineer:

  1. The five-stage pipeline model scales from 1 MB/day to 1 PB/sec. The same conceptual framework applies. CERN did not invent a new paradigm — they implemented the same stages at extreme scale.
  2. Ingestion filtering is unavoidable. At some volume, you cannot keep everything. You must make decisions about what to discard. CERN's trigger system is the most dramatic example, but every data engineer eventually faces the same question: "What data can I afford to drop?"
  3. Quality is a system property, not a step. CERN's data quality monitoring is embedded at every stage, not added at the end. The Higgs boson discovery was possible because data quality was engineered into the pipeline from the start.
  4. Orchestration at scale requires geographic distribution. Data lives where it is needed. Computation moves to the data, not the other way around. This principle applies to any global company (Uber, Netflix, Airbnb) as well as to CERN.
  5. Backfilling is inevitable. CERN re-processes petabytes of data every time calibration improves. Plan for it. Build idempotent transformations. Partition your data so that any time range can be re-processed independently.
The Ultimate Takeaway: The Large Hadron Collider proves that the data engineering mental model in this tutorial — Source → Ingestion → Storage → Transformation → Serving, wrapped in orchestration and data quality — is not just a pedagogical framework. It is a battle-tested architecture that powers the most ambitious scientific project in human history. Learn these principles, and you can build reliable data systems at any scale.
