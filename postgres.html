The Architecture You're Building

Your idea is a well-known pattern called thick database architecture. The database is not just a dumb storage layer — it's where business rules live. Here's what the stack looks like:

FRONTEND GitHub Pages — vanilla HTML/CSS/JS — fetch() calls Static
↕
CONTROLLER Thin Python / Node server — routes HTTP → calls DB functions Minimal
↕
SERVICE LAYER PostgreSQL stored procedures — business logic, validation, orchestration In Postgres
↕
REPOSITORY PostgreSQL functions + tables — raw data access, joins, inserts In Postgres
✓ The Golden Rule
Your controller should do ONE thing: receive an HTTP request, call a Postgres function, return the JSON result. No business logic outside the database.

The Project: Personal Bookshelf API

We're building a minimal bookshelf app. You can add books, mark them as read, and get your reading stats. It's small enough to finish in an hour but uses every layer of the architecture.

ℹ Why This Project
One entity (books), a few operations, one relationship (book → status). No noise. Maximum focus on the Postgres patterns you'll replicate in every future project.

Step 1 — Create Schemas

Schemas are your namespace. We'll use three: repo (data access), service (business logic), and api (public interface called by the controller).

SQL 
-- Create the three layers as schemas
CREATE SCHEMA IF NOT EXISTS repo;
CREATE SCHEMA IF NOT EXISTS service;
CREATE SCHEMA IF NOT EXISTS api;

-- Verify
SELECT schema_name FROM information_schema.schemas
WHERE schema_name IN ('repo', 'service', 'api');

Step 2 — Create the Table

Tables always live in repo. This is the data layer.

SQL 
CREATE TABLE repo.books (
  id          UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  title       TEXT          NOT NULL,
  author      TEXT          NOT NULL,
  status      TEXT          NOT NULL DEFAULT 'unread'
                            CHECK (status IN ('unread', 'reading', 'read')),
  rating      SMALLINT      CHECK (rating BETWEEN 1 AND 5),
  notes       TEXT,
  added_at    TIMESTAMPTZ   NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ   NOT NULL DEFAULT now(),
  finished_at TIMESTAMPTZ
);

-- Useful index for filtering by status
CREATE INDEX idx_books_status ON repo.books (status);

-- Confirm it exists
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_schema = 'repo' AND table_name = 'books';

Repository Layer — Functions

Functions are the repository layer. They know nothing about business rules — they just read and write data cleanly. In PL/pgSQL, a function always RETURNS something: a scalar, a row, or a table.

Return Types Cheat Sheet

Use CaseReturn TypeExample
Single valueRETURNS TEXTget user name
One rowRETURNS repo.booksget book by id
Multiple rowsRETURNS SETOF repo.bookslist books
Custom shapeRETURNS TABLE(...)stats query
JSON blobRETURNS JSONBAPI response
NothingRETURNS VOIDfire-and-forget update
Function Behavior Flags
STABLE tells Postgres the function reads data but doesn't modify it — same inputs always return the same result within a single statement. IMMUTABLE means results never change (even across statements). VOLATILE (default) means it can modify data or return different results on each call. Picking the right one lets the planner optimize queries.

Function 1 — Get All Books

PL/PGSQL 
CREATE OR REPLACE FUNCTION repo.get_all_books(
  p_status TEXT DEFAULT NULL   -- optional filter
)
RETURNS SETOF repo.books
LANGUAGE sql
STABLE   -- hint: no side effects, same result same inputs
AS $$
  SELECT *
  FROM repo.books
  WHERE (p_status IS NULL OR status = p_status)
  ORDER BY added_at DESC;
$$;

-- Test it
SELECT * FROM repo.get_all_books();
SELECT * FROM repo.get_all_books('read');

Function 2 — Get Book By ID

PL/PGSQL 
CREATE OR REPLACE FUNCTION repo.get_book(p_id UUID)
RETURNS repo.books
LANGUAGE sql STABLE AS $$
  SELECT * FROM repo.books WHERE id = p_id;
$$;

-- Test: grab a real id first
SELECT id FROM repo.books LIMIT 1;
-- Then: SELECT * FROM repo.get_book('paste-uuid-here');

Function 3 — Insert a Book

PL/PGSQL 
CREATE OR REPLACE FUNCTION repo.insert_book(
  p_title  TEXT,
  p_author TEXT,
  p_status TEXT DEFAULT 'unread'
)
RETURNS repo.books
LANGUAGE sql AS $$
  INSERT INTO repo.books (title, author, status)
  VALUES (p_title, p_author, p_status)
  RETURNING *;
$$;

-- Test it
SELECT * FROM repo.insert_book('Deep Work', 'Cal Newport');

Function 4 — Reading Stats (Custom Shape)

PL/PGSQL 
CREATE OR REPLACE FUNCTION repo.get_reading_stats()
RETURNS TABLE(
  total_books   BIGINT,
  books_read    BIGINT,
  currently_reading BIGINT,
  avg_rating    NUMERIC
)
LANGUAGE sql STABLE AS $$
  SELECT
    COUNT(*)                                           AS total_books,
    COUNT(*) FILTER (WHERE status = 'read')          AS books_read,
    COUNT(*) FILTER (WHERE status = 'reading')       AS currently_reading,
    ROUND(AVG(rating), 1)                             AS avg_rating
  FROM repo.books;
$$;

SELECT * FROM repo.get_reading_stats();
✓ Checkpoint
Your repo layer is complete. Every data operation is behind a named function. The service layer above it will never write raw SQL — it only calls these functions.

Service Layer — Stored Procedures & Logic

The service layer handles business rules: validation, orchestration, transactions, and side effects. This is where PL/pgSQL gets interesting — you get variables, conditionals, loops, and exception handling.

Anatomy of a PL/pgSQL Procedure

PL/PGSQL ANATOMY 
CREATE OR REPLACE PROCEDURE service.example(p_arg TEXT)
LANGUAGE plpgsql
AS $$
DECLARE
  v_variable  TEXT;           -- local variable (prefix v_)
  v_count     INTEGER := 0;  -- with default value
BEGIN
  -- Your logic here
  v_variable := p_arg || ' processed';

  IF v_count = 0 THEN
    RAISE EXCEPTION 'Nothing found';
  END IF;

EXCEPTION
  WHEN others THEN
    RAISE EXCEPTION 'Error: %', SQLERRM;
END;
$$;

Service 1 — Add Book (with validation)

Business rule: title must not be blank, and we don't allow duplicates by same title + author.

PL/PGSQL 
CREATE OR REPLACE FUNCTION service.add_book(
  p_title  TEXT,
  p_author TEXT,
  p_status TEXT DEFAULT 'unread'
)
RETURNS repo.books
LANGUAGE plpgsql AS $$
DECLARE
  v_title  TEXT := TRIM(p_title);
  v_author TEXT := TRIM(p_author);
  v_exists BOOLEAN;
BEGIN
  -- Validation
  IF LENGTH(v_title) = 0 THEN
    RAISE EXCEPTION 'Title cannot be blank'
      USING ERRCODE = '22023';
  END IF;

  -- Duplicate check
  SELECT EXISTS(
    SELECT 1 FROM repo.books
    WHERE LOWER(title) = LOWER(v_title)
      AND LOWER(author) = LOWER(v_author)
  ) INTO v_exists;

  IF v_exists THEN
    RAISE EXCEPTION 'Book already exists: % by %', v_title, v_author
      USING ERRCODE = '23505';
  END IF;

  -- Delegate to repo layer
  RETURN repo.insert_book(v_title, v_author, p_status);
END;
$$;

-- Test it
SELECT * FROM service.add_book('Atomic Habits', 'James Clear');
-- Try adding it again — should fail with duplicate error

Service 2 — Finish a Book (transaction)

When you finish a book, we update status AND record a timestamp. Atomic — both happen or neither does.

PL/PGSQL 
CREATE OR REPLACE FUNCTION service.finish_book(
  p_id     UUID,
  p_rating SMALLINT DEFAULT NULL
)
RETURNS repo.books
LANGUAGE plpgsql AS $$
DECLARE
  v_book repo.books;
BEGIN
  -- Fetch the book
  v_book := repo.get_book(p_id);

  IF v_book.id IS NULL THEN
    RAISE EXCEPTION 'Book not found: %', p_id;
  END IF;

  IF v_book.status = 'read' THEN
    RAISE EXCEPTION 'Book is already marked as read';
  END IF;

  -- Update atomically
  UPDATE repo.books
  SET
    status      = 'read',
    rating      = COALESCE(p_rating, rating),
    finished_at = now(),
    updated_at  = now()
  WHERE id = p_id
  RETURNING * INTO v_book;

  RETURN v_book;
END;
$$;

Trigger — Auto-update updated_at

The DEFAULT now() on the updated_at column handles INSERTs. But on UPDATE the column stays frozen unless explicitly set. This trigger fires before every UPDATE and stamps the current time automatically — no more manual updated_at = now() in your functions.

PL/PGSQL 
-- 1. Create the trigger function (reusable for any table)
CREATE OR REPLACE FUNCTION repo.set_updated_at()
RETURNS TRIGGER
LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at := now();
  RETURN NEW;
END;
$$;

-- 2. Attach it to the books table
CREATE OR REPLACE TRIGGER trg_books_updated_at
  BEFORE UPDATE ON repo.books
  FOR EACH ROW EXECUTE FUNCTION repo.set_updated_at();

-- Now updated_at is always accurate, automatically
✓ Checkpoint
Your service layer enforces business rules. The controller doesn't need to know about validation or duplicates — Postgres raises exceptions and the controller just returns them as HTTP errors.

Deep Dive — The API Layer (JSON Responses)

This is where everything clicks for your architecture. The api schema exposes functions that return complete JSON responses — the controller just passes the JSON blob straight to the HTTP client. Zero transformation needed.

Why JSONB in Postgres?

JSONB
Stored as binary. Supports indexing, operators, path queries. Use for data you query against.
JSON
Stored as text. Preserves whitespace. No indexing. Use only for exact round-trip storage.

Core JSON Operators

SQL 
-- row_to_json: turn a row into a JSON object
SELECT row_to_json(b) FROM repo.books b LIMIT 1;

-- json_agg: aggregate rows into a JSON array
SELECT json_agg(row_to_json(b)) FROM repo.books b;

-- jsonb_build_object: build a JSON object manually
SELECT jsonb_build_object(
  'success', true,
  'count', COUNT(*),
  'data', json_agg(row_to_json(b))
)
FROM repo.books b;

API Function — List Books

This is what the controller calls. It returns a complete, typed JSON envelope.

PL/PGSQL 
CREATE OR REPLACE FUNCTION api.list_books(
  p_status TEXT DEFAULT NULL
)
RETURNS JSONB
LANGUAGE plpgsql AS $$
DECLARE
  v_books JSONB;
  v_stats JSONB;
BEGIN
  -- Collect books as JSON array
  SELECT COALESCE(jsonb_agg(
    jsonb_build_object(
      'id',          b.id,
      'title',       b.title,
      'author',      b.author,
      'status',      b.status,
      'rating',      b.rating,
      'added_at',    b.added_at
    )
  ), '[]'::jsonb)
  INTO v_books
  FROM repo.get_all_books(p_status) b;

  -- Collect stats
  SELECT jsonb_build_object(
    'total',     s.total_books,
    'read',      s.books_read,
    'reading',   s.currently_reading,
    'avg_rating',s.avg_rating
  )
  INTO v_stats
  FROM repo.get_reading_stats() s;

  -- Return complete API envelope
  RETURN jsonb_build_object(
    'success', true,
    'data',    v_books,
    'meta',    v_stats
  );
END;
$$;

-- Test: full API response
SELECT api.list_books();
SELECT api.list_books('read');
Production Note
The EXCEPTION WHEN others catch-all below is concise for a demo, but in production you should discriminate error codes (e.g. WHEN unique_violation THEN) so the caller gets a proper 409 vs 500.

API Function — Add Book (with error envelope)

PL/PGSQL 
CREATE OR REPLACE FUNCTION api.add_book(
  p_payload JSONB   -- accept raw JSON from controller
)
RETURNS JSONB
LANGUAGE plpgsql AS $$
DECLARE
  v_book repo.books;
BEGIN
  v_book := service.add_book(
    p_title  := p_payload->>>'title',
    p_author := p_payload->>>'author',
    p_status := COALESCE(p_payload->>>'status', 'unread')
  );

  RETURN jsonb_build_object(
    'success', true,
    'data',    row_to_json(v_book)
  );

EXCEPTION
  WHEN others THEN
    RETURN jsonb_build_object(
      'success', false,
      'error',   SQLERRM,
      'code',    SQLSTATE
    );
END;
$$;

-- Test it
SELECT api.add_book('{"title":"Zero to One","author":"Thiel"}'::jsonb);
-- Error case
SELECT api.add_book('{"title":"","author":"Nobody"}'::jsonb);

The Thin Controller (Python example)

This is ALL the Python controller does — one connection, zero business logic:

Production Note
This demo uses a single persistent connection. In production, use a connection pool (e.g. psycopg2.pool or PgBouncer) so each request gets its own connection from the pool.
PYTHON 
# requirements: fastapi, psycopg2-binary, uvicorn
from fastapi import FastAPI
import psycopg2, json

app = FastAPI()
conn = psycopg2.connect("dbname=bookshelf user=postgres")

@app.get("/books")
def list_books(status: str = None):
    with conn.cursor() as cur:
        cur.execute("SELECT api.list_books(%s)", [status])
        return cur.fetchone()[0]   # already JSON, just return it

@app.post("/books")
def add_book(payload: dict):
    with conn.cursor() as cur:
        cur.execute("SELECT api.add_book(%s)", [json.dumps(payload)])
        return cur.fetchone()[0]   # already JSON, just return it

The controller is 20 lines. It does zero business logic. It has zero validation. Everything lives in Postgres. Your GitHub Pages frontend calls these endpoints and renders the JSON. That's the full stack.
